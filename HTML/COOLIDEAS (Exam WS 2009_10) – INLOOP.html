COOLIDEAS (Exam WS 2009/10)

There is an online platform for innovation management to be developed at the university. They decided to use the ‘Open Innovation’ concept. The main idea is that organizations in a world of widely distributed knowledge cannot rely on their own innovation force alone. Instead, the integration and use of external information and competences become more and more important.

COOLIDEAS should support the organization of a community of ‘innovators’: Ideas of students, employees, professors and external partners as well are exchanged, discussed and evaluated. In the future, it should be possible to realize a comprehensive idea management processes.

To distinguish between the implementation and analysis classes, all implementation class names begin with ‘J‘ (like implementation with Java).

Class Diagram

The class diagram below shows the classes needed for the idea management. See below for further explanations of the different aspects of this large structure.

Idea Pool

The class JIdeaPool should manage all topics and the ideas belongig to them.

  • The attribute pool must be implemented as a java.util.Map, which maps each JTopic to the java.util.Set of JIdeas belonging to that JTopic. Note that a JIdea may belong to several JTopics.
  • The add() methods add either a topic or an idea and its topic to the pool of ideas. A JIdeaPool may not contain several ideas with the same title.
  • The remove() methods remove an idea or a topic from the pool. If the idea or the topic does not exist, remove() returns false, otherwise true. Make sure to remove an idea from every topic, if it is associated to multiple topics!
  • The numberOfTopics() method returns the number of different topics existing in the pool of ideas. The method numberOfIdeas() works analogous.
  • The getIdea() method returns the idea with the given title. If there is no idea with the given title, null must be returned.

The output of the toString() method should look like this:

Content: Title
Description

Of course, Content must be replaced by the appropriate tag: Attachment, Idea, Topic or Valuation.

Idea Exchange Process

Now, we take a closer look at the usage of the state pattern in the class structure. In this pattern, we use an abstract state class (JIdea.JState in this case) which provides the design of all concrete states. Whenever an undefined transition between states is triggered, an IllegalxStateException must be thrown. If you implement this behavior in the JIdea.JState class you do not need to override the methods in each concrete state class. To understand which transition is allowed in which state, take a look at the following state diagram.

  • Once declined or released, an JIdea‘s state cannot be changed anymore. All declined or released ideas can be removed from the pool with the methods JIdeaPool.removeReleased() and JIdeaPool.removeDeclined().
  • The evaluate() method should change the valuation attribute.
  • The discuss() method appends the given discussion String to the previous discussions (separated by a newline character).

Remember to implement the state pattern the way it was introduced in the lecture: All states must be inner classes of the context class.

Getting Updated on Topics

Every member of the community (JMember) should be able to subscribe to JTopics to be notified whenever they are changed. Therefore, we extend our design like this:

  • topics should be a java.util.Set
  • Whenever the title or description of a JTopic is changed, the methods setChanged() and notifyObservers(Object arg) (both inherited from Observable) should be called. The value specified with the arg parameter is passed to each Observer observing the JTopic as the arg1 argument of their method update(Observable arg0, Object arg1).
  • JMember.update(...) should print the following message to the console: "The topic <id> has been updated!", where <id> is replaced with the concerning topic’s id. Make sure to use System.out.println() or the tests will fail!

Task

Implement the idea management classes, the state pattern’s behavior and the observer functionality!

Hints

  • The most important erroneous parameter values must be handled properly. Therefore, the parameter value null should result in a NullPointerException while others like empty Strings should cause an IllegalArgumentException.
